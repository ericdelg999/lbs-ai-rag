# Bulbrite AI Product Answer Copilot (POC)
**PRD + Technical Implementation Reference (Combined)**

## 1) Project Overview

### Project Name
Bulbrite AI Product Answer Copilot (POC)

### Owner
Eric (Ecommerce/Ops Lead)

### Purpose
Build an internal AI-powered product-answer and quote-assist tool for **Bulbrite products only** (POC) that helps customer service and sales reps quickly answer product questions using:
- BigCommerce product export data
- BigCommerce custom fields (parsed from raw custom field string)
- Bulbrite spec sheets (downloaded from LBS-hosted spec sheet URLs)
- Optional inventory later (out of scope for initial POC)

### Why This Project
This project is intended to:
- Deliver a high-value internal tool (similar leverage to the AI upload system)
- Practice working with multiple APIs and retrieval-based AI architecture (RAG pattern)
- Create a reusable framework that can scale to other brands with cleaner data
- Serve as a strong project for PRD / Jira / sprint planning practice

---

## 2) Problem Statement

CS and sales reps currently need to manually search:
- Product pages
- Spec sheets
- Custom fields / technical details
- Possibly inventory or availability sources

This creates slow response times, inconsistent answers, and difficult training for new reps.

### Core Problem
There is no internal tool that can answer natural-language product questions across structured product data + spec sheets in one place with evidence-backed responses.

---

## 3) Goals and Non-Goals

## Goals (POC)
1. Build a working internal app (Streamlit is acceptable) for **Bulbrite-only** product Q&A.
2. Support common CS/sales natural-language queries and return useful SKU matches.
3. Use both structured BigCommerce export data and spec sheet text for retrieval.
4. Return answers with supporting evidence/snippets (not just AI-generated text).
5. Build architecture in a **multi-brand-ready** way even though only Bulbrite is deployed.

## Non-Goals (POC)
1. Full multi-brand support (Bulbrite only for initial POC).
2. Live inventory integration (can be added later).
3. Teams/Gorgias integration (later phase).
4. Automated quoting/cart creation (later phase).
5. Perfect product compatibility logic (initially focus on retrieval + product-answering).

- Customer service reps (internal)

## 5) User Jobs to Be Done

1. **Find products matching technical criteria**
   - Example: “I need a Bulbrite 3500K recess disk under $50 and at least 9W.”

2. **Answer spec questions quickly**
   - Example: “Is this model dimmable?”
   - Example: “What base type is this lamp?”

3. **Find replacements / similar options**
   - Example: “What Bulbrite option is similar to SKU X?”

4. **Reduce time spent opening multiple PDFs/PDPs**
   - Return best candidates + supporting specs in one place.

## 8) Key Data Sources (Bulbrite POC)

## 8.1 BigCommerce Export (Primary Structured Source)
Current known available/exportable data includes:
- H1 / product title
- Product description
- Custom fields (exported as one raw string like `"Field=Value";"Field2=Value2";...`)
- Price (if included / derivable)
- PDP URL (can be generated)
- SKU
- Additional standard product metadata depending on export template

### Notes
- **Technical facts should prioritize product/row/custom-field data over spec text** when conflicts occur.
- For Bulbrite, this is likely clean enough for strong POC performance.

## 8.2 Spec Sheets (Secondary Supporting Source)
- Hosted on LBS and generally follow pattern:
  - `https://lightbulbsurplus.com/content/BULR-SKU.pdf` (example pattern; exact format to confirm in code)

- WebDAV (Cyberduck) exists as a fallback/source of truth

## 8.3 Inventory (Deferred)
- Out of scope for initial POC

## 9) Technical Concept (RAG Pattern Clarification)

## What RAG Means in This Project
RAG (Retrieval-Augmented Generation) is a system pattern where:
1. Product knowledge is stored in searchable indexes (not inside the LLM)
2. Data is chunked + embedded for semantic retrieval
3. User query retrieves top relevant chunks/records
4. Retrieved context is sent to an API model to generate a grounded answer

## Important Clarification
- The AI model (OpenAI/Azure OpenAI/etc.) does **not** automatically know or access the catalog data.
- The app code performs retrieval and passes relevant context into the model request.

## 10) Source Priority Rules (Critical)

When the same technical fact appears in multiple places and conflicts exist, use this priority:

### Technical Attributes (highest confidence first)
1. **Parsed structured custom fields / row data**
2. **Spec sheet text**
3. **Product description**
4. **H1/title**

### Marketing / use-case language
- Product description + spec features can be used for richer explanatory answers

### Why this matters
This prevents spec-sheet combo/matrix ambiguity (especially relevant for future multi-brand scaling).

## 11) Canonical Data Model (POC)

The POC does **not** require fully exploding every custom field into separate CSV columns.  
However, custom fields should be parsed into structured JSON for retrieval and normalization.

## 11.1 Core Product Record (Canonical Product Object)
Each SKU should produce a canonical product object with at least:

- `sku`
- `brand` (Bulbrite)
- `product_name_h1`
- `description_html` (or text-cleaned version)
- `custom_fields_raw` (original BigCommerce CF string)
- `custom_fields_json` (parsed dict)
- `pdp_url`
- `spec_sheet_url`
- `price` (if available)
- `active_status` (if available)
- `category` / `product_type` (if available)

## 11.2 Canonical Normalized Technical Fields (Subset for Filtering/Ranking)
Extract when possible from `custom_fields_json` (or fallback sources):
- `wattage_actual`
- `lumens_actual`
- `voltage`
- `color_temperature` / `cct`
- `base_type`
- `shape`
- `dimmable`
- `finish` (for fixtures)
- `pack_qty`
- `bulb_or_fixture_type`

### Notes
- Not all fields will exist for all products.
- POC can proceed with missing values.
- Store both raw and normalized forms for traceability/debugging.

## 12) Chunking Strategy (POC)

## Overview
Do **not** rely only on raw PDF chunks.
Use a mixed chunk strategy that prioritizes structured product data.

## 12.1 Chunk Type A — SKU Master Record (Most Important)
One chunk per SKU containing:
- SKU
- Brand
- H1/title
- Short/clean description text
- Flattened custom field values (parsed CF JSON rendered to text)
- Price (if available)
- PDP URL
- Spec sheet URL
- Product type/category (if available)

### Purpose
This chunk should answer many common CS/sales questions with cleaner data than PDF text.

## 12.2 Chunk Type B — Spec Sheet Chunks (1 to N per SKU)
Spec sheet text split into logical sections when possible:
- Features / overview
- Electrical
- Performance (lumens/CCT/CRI/life)
- Dimensions / physical
- Compliance / ratings
- Ordering info / model matrix (may be noisy but useful)

### Fallback
If section headers are inconsistent, use token-length chunking with overlap.

## 12.3 Chunk Metadata (Required on Every Chunk)
Each chunk should include metadata such as:
- `chunk_id`
- `sku`
- `brand`
- `doc_type` (`sku_record` or `spec_sheet`)
- `source_url`
- `product_type` (if available)
- `price` (optional)
- `active_status` (if available)
- `source_priority` (for conflict/debug logic)
- `chunk_label` (e.g., `electrical`, `performance`, `overview`)

---

## 13) Search / Retrieval Strategy (POC)

## Required Retrieval Behavior
Use a **hybrid retrieval strategy** whenever possible:
1. **Keyword/exact matching** (for SKUs, model numbers, exact terms)
2. **Vector/semantic search** (for natural language queries)
3. **Metadata filtering** (brand=Bulbrite, optional product type, price constraints later)

### Why Hybrid Matters
Technical catalogs contain:
- SKUs / part numbers (best with exact match)
- Natural language needs (best with vector search)

## Query Handling Expectations (POC)
- If the query is non-brand-specific, POC should still return the best matches from **Bulbrite only**
- UI should make POC scope clear (e.g., “Showing Bulbrite results”)
- Later versions can add multi-brand routing + brand preference rules

## Future Brand Preference Logic (Out of Scope for v1, design-ready)
Open-ended query ranking may later include:
1. Hard filters (price / specs)
2. Brand priority order
3. Semantic ranking among filtered candidates

## 15) Model Roles (POC)

## 15.1 Embeddings Model
Used for:
- Precomputing chunk embeddings (indexing time)
- Embedding user queries (query time)

## 15.2 Answer Model (Chat/Reasoning Model)
Used for:
- Synthesizing final response from retrieved chunks + structured product results
- Explaining why specific SKU(s) match
- Returning grounded answer with evidence snippets

## Practical POC Guidance
- Start with **one embeddings model** and **one cost-efficient answer model**
- Retrieval quality is usually the bottleneck before model quality
- Add stronger answer model only if needed after testing

## 16) Script-by-Script Breakdown (POC Implementation)

This section defines a modular pipeline for ingestion, indexing, querying, and UI.

---

## Script 1 — `export_prep_bulbrite.py`
### Purpose
Prepare BigCommerce export data into a cleaner intermediate dataset for Bulbrite products.

### Inputs
- BigCommerce export CSV (full or Bulbrite-filterable)

### Outputs
- `bulbrite_products_prepped.csv` (or `.jsonl`)

### Responsibilities
- Load BigCommerce export
- Filter to Bulbrite products
- Preserve raw key fields (SKU, H1, description, custom field string, price, etc.)
- Build/generate:
  - `pdp_url`
  - `spec_sheet_url` (from SKU pattern)
- Basic cleanup (trim whitespace, normalize empty values)
- Log row counts and exclusions

### Model Usage
- None

---

## Script 2 — `parse_custom_fields.py`
### Purpose
Parse BigCommerce custom field raw string into structured JSON and extract a normalized canonical subset.

### Inputs
- `bulbrite_products_prepped.csv`

### Outputs
- `bulbrite_products_parsed.csv` (or `.jsonl`) with:
  - `custom_fields_raw`
  - `custom_fields_json`
  - canonical normalized subset fields (where found)

### Responsibilities
- Parse strings like:
  - `"Field=Value";"Field2=Value2";...`
- Handle inconsistent quotes / malformed formatting gracefully
- Store parsed fields in JSON/dict format
- Normalize common field aliases (e.g., `Color Temp` vs `Color Temperature`)
- Extract canonical subset (wattage, lumens, CCT, base, shape, dimmable, etc.)
- Preserve all raw data for traceability

### Model Usage
- None (rule-based parser only)

### Notes
- **Do not** create a separate CSV column for every unique custom field for v1 unless needed
- A JSON column/object is acceptable and preferred for POC flexibility

---

## Script 3 — `download_spec_sheets.py`
### Purpose
Download Bulbrite spec sheet PDFs using generated LBS spec sheet URLs.

### Inputs
- `bulbrite_products_parsed.csv` (or prepped file with `spec_sheet_url`)

### Outputs
- Local PDF folder (e.g., `data/spec_pdfs/`)
- Download log (success/failure/missing)

### Responsibilities
- Download spec sheets from generated URLs
- Save files using deterministic naming (e.g., `SKU.pdf`)
- Handle retries/timeouts
- Log failures for later retry
- Optional fallback to WebDAV path (future enhancement if needed)

### Model Usage
- None

---

## Script 4 — `extract_spec_text.py`
### Purpose
Extract text from spec sheet PDFs using existing extraction logic with OCR fallback.

### Inputs
- Local PDF folder (`data/spec_pdfs/`)

### Outputs
- Extracted text files or JSON files per SKU (e.g., `data/spec_text/`)
- Extraction metadata log

### Responsibilities
- Attempt standard PDF text extraction first
- OCR fallback for scanned/low-text PDFs
- Save extracted text per SKU
- Save metadata:
  - extraction method used
  - pages processed
  - OCR fallback used (true/false)
  - errors/warnings

### Model Usage
- None (existing OCR/text extraction logic reused)

### Important Note
For future multi-brand expansion with combo spec sheets, product row data remains higher priority than spec text.

---

## Script 5 — `build_chunks.py`
### Purpose
Create chunk objects for both SKU master records and spec sheet content.

### Inputs
- `bulbrite_products_parsed.csv`
- Extracted spec text files

### Outputs
- `chunks.jsonl` (one chunk per line)

### Responsibilities
- Build **SKU Master Record chunk** per SKU
- Build **Spec Sheet chunks** (section-based or token-based)
- Attach metadata to every chunk
- Assign `doc_type`, `chunk_label`, `source_priority`, etc.
- Keep chunk text clean and deterministic for embedding/indexing
- Track chunk counts per SKU

### Model Usage
- None

### Example Chunk Types
- `doc_type=sku_record`
- `doc_type=spec_sheet`

---

## Script 6 — `embed_and_index.py`
### Purpose
Generate embeddings for chunks and store them in the vector index; store structured product table for display/filtering.

### Inputs
- `chunks.jsonl`
- Parsed product records (for SKU table)

### Outputs
- Vector index population
- Structured product DB/table (SQLite for POC)
- Indexing logs

### Responsibilities
- Read chunk records
- Call embeddings API in batches
- Store:
  - chunk text
  - embedding vector
  - metadata
- Store structured SKU table in SQLite (or similar)
- Maintain idempotent behavior if rerun (optional but recommended)

### Model Usage
- Embeddings API model only

---

## Script 7 — `query_service.py` (core retrieval/orchestration module)
### Purpose
Handle retrieval, ranking, and answer-generation orchestration for each user query.

### Inputs (runtime)
- User query (natural language)
- Optional filters (brand/product type/price cap, etc.)

### Outputs (runtime)
- Final answer text
- Top SKU results
- Evidence snippets / source chunks
- Retrieval/debug metadata (optional internal)

### Responsibilities
- Parse user query for:
  - SKU mentions
  - obvious structured constraints (price, wattage, color temp, brand mention)
- Apply POC scope filter (`brand=Bulbrite`)
- Run hybrid retrieval:
  - keyword exact search (SKU/model)
  - vector similarity search
- Merge/re-rank top results
- Select top chunks + corresponding SKU rows
- Build model prompt/context package
- Call answer model
- Return grounded response with evidence

### Model Usage
- Embeddings model (query embedding)
- Answer model (chat/reasoning)

### Notes
- Retrieval and ranking logic should live here (not in Streamlit UI)
- Brand-preference routing can be added later here for multi-brand support

---

## Script 8 — `app_streamlit.py`
### Purpose
Provide an internal user interface for the Bulbrite POC.

### Inputs
- User-entered question
- Optional UI filters

### Outputs
- Answer panel
- SKU result cards
- Evidence snippets
- Feedback capture

### Responsibilities
- Render input box for natural-language query
- (Optional) render filters (price cap, product type, etc.)
- Call `query_service.py`
- Display:
  - grounded answer
  - top matching SKUs
  - PDP/spec links
  - evidence snippets
- Capture feedback:
  - Helpful
  - Wrong
  - Missing info

### Model Usage
- None directly (via service layer)

---

## Script 9 — `eval_runner.py` (recommended)
### Purpose
Run a repeatable evaluation set against the POC before internal pilot.

### Inputs
- Test question set (20–50 real Bulbrite CS/sales questions)
- Query service access

### Outputs
- Evaluation results file (CSV/JSON)
- Latency and retrieval logs
- Manual scoring worksheet (optional)

### Responsibilities
- Run each test question through `query_service.py`
- Save:
  - answer text
  - returned SKU results
  - evidence chunks
  - timing
- Support manual scoring and failure analysis
- Help prioritize fixes before pilot

### Model Usage
- Same as query service (indirect)
